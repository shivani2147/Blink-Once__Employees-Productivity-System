Build a complete production-style web application called "Blink Once Employees Productivity System" using Python + FastAPI + HTML + CSS + SQL Server.

Technical Requirements : 
Backend: FastAPI
Frontend: HTML, CSS
Database: Microsoft SQL Server
ORM: SQLAlchemy
Authentication: Secure login with password hashing using bcrypt/passlib
Database driver: pyodbc
Form handling using FastAPI
Use a modular project structure
Include proper validation
Include session-based authentication

Create a config.py file that contains any location paths, SQL connection configuration, etc:

DATABASE = "BlinkOnce_EmployeesProductivity"

Employee Dashboard

Create form with these fields:
Employee name, Designation, Client name, Date, Project name, project Start & end date, Video Duration, Status (Ongoing, Hold, Not Started, Done), Harddisk number, Directory/location in harddisk, Uploaded to drive (Yes/No), Drive Link, Type of shoot (engagement, wedding, pre-wedding, corporate, birthday, events,.), Number of Cameras Used, Comments

Employee can create records
Employee can edit own records
Employee can view own records
Employee cannot view other employees' records

"Admin Dashboard"

Admin requirements:
Admin should ONLY have viewing permission.

Admin can:
View all employees, View all project/productivity entries
Filter by --> Employee, Client Name, Project Name, Status, Date range

Admin dashboard should display:
Total Employees, Total Projects, Completed Projects, Ongoing Projects, Hold Projects, Progress work of each employee, Employee Productivity Camera-Level Analytics Graph
Use dashboard cards.

Requirements for Camera-Level Analytics:
Create a chart/graph showing each employee's workload and productivity based on camera levels.
Camera Levels should be categorized as:
Level 1 = Single Camera footage
Level 2 = Two Camera footage
Level 3 = Three Camera footage
Level N = N number of cameras

The system should automatically calculate the expected editing workload and estimated completion time according to the number of cameras involved.

The password must be hashed before storing it in SQL Server.

Admin Identification & Access Control:

Use Role-Based Access Control (RBAC) with a `role` field in the database (Employee/Admin). On first application startup, if no Admin account exists in the database, automatically create a default Admin account:
Username: admin
Password: admin123
Role: Admin
Store admin credentials in the database with hashed passwords, and prevent privilege escalation or direct URL access to admin pages.

Admin should have the ability to change their username and password from the profile/settings page. Any credential changes must automatically update the SQL Server database. Passwords must always be hashed before storage. After changing credentials, future logins should use the updated credentials from the database instead of the default ones.
